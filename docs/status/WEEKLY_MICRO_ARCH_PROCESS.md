# Vilsay · 每周微架构门禁流程

**版本：** 1.0 | **日期：** 2026-03-22  
**适用：** 领取或开工任意 **Week N** 开发任务之前  
**参与角色：** **开发**（实现方）与 **架构**（设计与接口负责人）须**共同**完成；单方不可代签。

---

## 1. 目的

避免任务书条目过于笼统导致的实现漂移、验收争议与集成期集中爆雷。在每周任务被**正式领取**或**标为完成**之前，对「本周要动哪些模块、边界在哪、失败时怎么表现」达成**书面一致**。

---

## 2. 触发时机

| 时机 | 动作 |
|------|------|
| **Week N 开工前** | 完成本章第 3 节清单，产出 `WeekN_MICRO_ARCH.md`（见第 4 节路径） |
| **Week N 收尾前** | 对照清单做「实现 vs 文档」差异修订；若有偏离，更新 `VILSAY_TECH_ARCH.md` 或任务书备注 |

---

## 3. 必须对齐的内容（清单）

开发 + 架构须就以下项逐项**确认并写入**当周微架构文档：

1. **范围**：本周任务编号（如 W4-02～W4-04）与代码目录/模块对应关系（`Core/`、`Entry/` 等）。
2. **对外契约**：新增或变更的 API（Swift `public`/`internal` 类型、异步边界、主线程约束）。
3. **数据流**：输入/输出、错误传播路径、`AppState` / `Pipeline` 是否受影响。
4. **权限与沙盒**：是否涉及麦克风、辅助功能、网络、文件、剪贴板；Entitlements 是否需变更。
5. **失败与降级**：超时、无网、权限被拒、第三方 API 非 2xx 时的用户可见行为（菜单提示、`lastPipelineError` 等）。
6. **与上游文档关系**：若实现与 `VILSAY_TECH_ARCH.md` / `VILSAY_DEV_TASKS.md` 不一致，**先改文档再合码**，或在该周微架构文档中明确「有意偏离」及原因。
7. **验收步骤**：可执行的 3～10 条本机操作步骤（含前置条件，如已配置 API Key、已开辅助功能）。

---

## 4. 落盘位置与命名

- 目录：`docs/status/micro-arch/`
- 文件名：`Week1_MICRO_ARCH.md`、`Week2_MICRO_ARCH.md` … 与任务书 Week 编号一致。
- **Week 1～3  retrospectively**：若当时未写，可补一份「追认版」，标注日期与「追认」字样，便于审计。

---

## 5. 门禁规则（建议写入团队规范）

- **未**产出当周 `WeekN_MICRO_ARCH.md` 或未通过第 3 节清单评审 → **不**将本周任务标为「可领取」或「全部 ✅」。
- 架构变更（模块拆分、协议替换）必须反映在 `docs/spec/VILSAY_TECH_ARCH.md` 的对应章节，并在变更记录表登记。
